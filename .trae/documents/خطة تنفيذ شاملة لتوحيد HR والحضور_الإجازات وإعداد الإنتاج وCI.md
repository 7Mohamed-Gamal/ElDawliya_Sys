## الأهداف الرئيسية
- تنظيف وتوحيد منظومة الموارد البشرية، الحضور، والإجازات، مع إزالة الازدواجية وطبقة توافق مرحلية.
- استكمال القوالب والعروض لضمان عمل جميع صفحات HR دون أخطاء.
- توحيد النماذج والروابط بين Attendance/Leaves/Payrolls/Employees مع اعتماد `employees.Employee` كمصدر بيانات أساسي.
- تجهيز إعدادات وأمن الإنتاج (أسرار، CORS/CSRF، تسجيلات، Celery) مع بيئات منفصلة.
- إصلاح CI وإضافة اختبارات تغطي المسارات الأساسية، مع مؤشرات قياس واضحة.

## الأهداف الفرعية
- توحيد نموذج الإجازات وإيقاف اعتماد النموذج المكرر تدريجيًا بطبقة خدمة توافق.
- توحيد قواعد الحضور باعتماد طبقة واحدة أو محول خدمة للمخطط القديم.
- إصلاح الاستيرادات غير الصحيحة في الخدمات.
- تحديث مسارات Celery إلى مهام متاحة فعليًا فقط.
- ضبط المتغيرات الحساسة عبر `.env` وتفعيل سجلات آمنة.

## الموارد المطلوبة
- بشرية: 
  - مسؤول تقني (Tech Lead) للتنسيق.
  - مطور Backend Django (2) لتوحيد النماذج والخدمات.
  - مطور Frontend/قوالب (1) لإتمام القوالب وتجانس UI.
  - مهندس DevOps (1) لضبط بيئات، Celery، وCI/CD.
  - مختبر QA (1) لتصميم وتنفيذ واختبارات قبول.
  - مسؤول قاعدة بيانات (DBA) (اختياري) لتدقيق الترحيل والأداء.
- مادية:
  - خادم اختبار/Stage لقاعدة بيانات SQL Server.
  - أجهزة حضور ZK للاختبار إن أمكن (أو محاكيات).
- تقنية:
  - Redis لوسيط Celery ونتائج.
  - GitHub Actions/Codecov للـ CI والتغطية.
  - أدوات مراقبة سجلات (Sentry/Logging).

## مؤشرات الأداء (KPIs) لكل مرحلة
- المرحلة 1 (توحيد النماذج):
  - 100% إزالة استخدام `attendance.LeaveType` في الطبقات الجديدة، واستبدالها بـ `leaves.LeaveType` عبر الخدمة.
  - نجاح اختبارات التوافق: ≥ 95% تمرير دون أخطاء.
  - زمن استعلامات الإجازات بعد التوحيد لا يزيد عن baseline ±10%.
- المرحلة 2 (القوالب والواجهة):
  - اكتمال القوالب المفقودة (4/4) وتشغيل العروض دون أخطاء (0 أخطاء 500/404).
  - رضا المستخدم الداخلي ≥ 80% في مراجعة UI.
- المرحلة 3 (تكامل Attendance/Leaves/Payrolls):
  - اعتماد مصدر واحد لـ `Leave` في الحضور والرواتب.
  - دقة خصومات الرواتب مقارنةً ببيانات الحضور: انحراف ≤ 2% في عينات اختبار.
- المرحلة 4 (إعدادات وأمن الإنتاج):
  - 100% نقل الأسرار إلى `.env` وإزالة القيم الصلبة.
  - تفعيل CORS/CSRF بشكل صحيح مع 0 حوادث رفض غير مبرر.
  - سجلات منظمة مع معدل أخطاء حرجة < 1% أسبوعيًا.
- المرحلة 5 (اختبارات وCI/CD):
  - تغطية الاختبارات ≥ 75% للوحدات الأساسية (HR/Attendance/Leaves/Payrolls).
  - زمن بناء CI < 8 دقائق، نجاح السير في أول تشغيل ≥ 90%.
- المرحلة 6 (الاستعداد للإنتاج):
  - تشغيل Celery worker/beat بنجاح، المهام المجدولة تعمل ≥ 95% من الدورات دون فشل.
  - فحوصات صحة (health checks) مستقرة لمدة 7 أيام.

## المخاطر وخطط الطوارئ
- مخاطر ترحيل البيانات بين نماذج الإجازات: 
  - خطة: طبقة توافق + ترحيل تدريجي + نسخ احتياطي قبل التنفيذ + إمكانية rollback.
- تعطل صفحات بسبب تغييرات القوالب:
  - خطة: تشغيل بيئة staging، فحص آلي وروبوتي للعروض، اختبار قبول يدوي قبل الدمج.
- فشل مهام Celery أو ازدواج مسارات:
  - خطة: توحيد المسارات، اختبار وحدات المهام، مراقبة عبر لوغ/flower، آلية إعادة محاولة مع حدود.
- تسريب أسرار أو إعدادات خاطئة:
  - خطة: فحص أمني للـ `.env`, مراجعة أذونات، استخدام vault اختياري.
- أداء قواعد الحضور:
  - خطة: قياس baseline قبل وبعد، تحسين فهارس، مسح تحليلي.

## المتابعة والتقييم الدوري
- اجتماعات أسبوعية سريعة (30 دقيقة) لعرض تقدم المراحل وKPIs.
- لوحة مؤشرات داخلية تجمع: نجاح المهام المجدولة، تغطية الاختبارات، أخطاء السجلات.
- تقارير مرحلية عند نهاية كل مرحلة تتضمن مؤشرات، مشكلات، وإجراءات تصحيح.

## التوثيق المطلوب
- وثائق قرار معماري (ADR) لتوحيد النماذج وقواعد الحضور.
- دليل إعداد البيئات (`.env`، Celery، Redis، SQL Server).
- توثيق واجهة الخدمة الموحّدة للإجازات والحضور.
- سجل تغييرات (Changelog) لكل مرحلة.
- Runbooks للتشغيل ومواجهة الأعطال.

## خطة التنفيذ المرحلية مع المهام والجدول
- المرحلة 0: التحضير (3 أيام)
  - جمع baseline للأداء والاعتماديات، إعداد بيئة staging، مراجعة الملفات الحرجة.
  - تسليم: تقرير جاهزية + خطة ترحيل بيانات عالية المستوى.
- المرحلة 1: تنظيف الموارد البشرية وتوحيد النماذج (5 أيام)
  - إنشاء طبقة خدمة توافق لربط `attendance.LeaveBalance/LeaveType` بـ `leaves.EmployeeLeave/LeaveType`.
  - تحديث الاستدعاءات في العروض/النماذج لاستخدام الواجهة الموحدة.
  - تسليم: طبقة خدمة موحّدة + اختبارات وحدة وتمرير ≥ 95%.
- المرحلة 2: القوالب والواجهة (4 أيام)
  - استكمال القوالب المفقودة، توحيد التصميم نحو `base.html`، نقل العناصر المشتركة إلى `components` عند الحاجة.
  - تسليم: تشغيل جميع صفحات HR بدون أخطاء + تقرير قابلية الاستخدام الداخلي.
- المرحلة 3: التكامل Attendance/Leaves/Payrolls (4 أيام)
  - توحيد مصدر بيانات الإجازات للحضور والرواتب، تحديث حسابات الرواتب لخصومات تعتمد `AttendanceSummary`.
  - تسليم: حسابات رواتب متسقة مع الحضور + اختبارات تكامل.
- المرحلة 4: إعدادات وأمان الإنتاج (3 أيام)
  - نقل الأسرار إلى `.env`, ضبط `ALLOWED_HOSTS` و`CSRF_TRUSTED_ORIGINS` وCORS، تنظيم التسجيلات.
  - تحديث مسارات Celery للمهام المتاحة فقط، تفعيل الجدولة الضرورية.
  - تسليم: بيئة إنتاج آمنة + فحوصات أمنية أولية.
- المرحلة 5: الاختبارات وCI/CD (3 أيام)
  - إصلاح CI لتشغيل اختبارات Django مع تغطية، إضافة اختبارات لوحات HR/Attendance/Leaves/Employees.
  - تسليم: سير CI أخضر + تغطية ≥ 75%.
- المرحلة 6: الاستعداد للإنتاج والتشغيل (3 أيام)
  - تشغيل العامل والمجدول، اختبار مزامنة الأجهزة، إعداد health check ومراقبة.
  - تسليم: تشغيل مستقر لمدة 7 أيام في staging مع تقارير صحة.
